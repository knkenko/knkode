# Security Audit: PR #13 — Canvas Terminal Renderer

## Summary

The canvas-based terminal renderer and keyboard-to-ANSI converter have a low overall attack surface because canvas `fillText` does not parse HTML and the data flow is well-structured (Rust emits pre-rendered `GridSnapshot` objects, not raw escape sequences). The main risks are CSS font-injection via the unsanitized `fontFamily` prop, and a potential denial-of-service through unbounded grid rendering. No XSS, prototype pollution, or unsafe innerHTML/eval patterns were found.

## Must Fix

- **`src/components/CanvasTerminal.tsx:48` / `src/components/CanvasTerminal.tsx:161` -- CSS font-string injection via `fontFamily` prop**: The `fontFamily` value is interpolated directly into `ctx.font` template literals (e.g., `` `${fontSize * dpr}px ${fontFamily}` ``). While this is Canvas2D (not CSS style injection in the DOM sense), a malicious `fontFamily` string could include additional font descriptors or break the font shorthand syntax in unexpected ways. More critically, at line 235, the `background` prop is passed into an inline `style={{ background }}` attribute on the container div. If these props originate from user-editable theme data (which `PaneTheme.fontFamily` and `PaneTheme.background` appear to be), a crafted value like `"red; position:fixed; z-index:9999"` could inject arbitrary CSS through the `background` style prop. **Recommendation**: Validate `fontFamily` against an allowlist of known monospace font stacks, or at minimum strip semicolons, braces, and non-alphanumeric/comma/space/quote characters. For `background`, validate it is a valid hex color or CSS color keyword before applying.

## Suggestions

- **`src/components/CanvasTerminal.tsx:74-85` -- No bounds checking on grid dimensions**: The draw loop iterates over `g.rows` without any upper-bound guard. If Rust sends a malformed or excessively large `GridSnapshot` (e.g., thousands of rows/columns), this will cause the renderer to freeze the UI thread with an O(rows * cols) synchronous canvas paint. **Recommendation**: Add a sanity check capping the rendered grid to a reasonable maximum (e.g., `Math.min(g.rows.length, 500)` rows and a column cap) and log a warning if exceeded. This serves as defense-in-depth against a misbehaving backend or corrupted IPC payload.

- **`src/components/CanvasTerminal.tsx:88-107` -- No bounds checking on cursor position**: `g.cursorRow` and `g.cursorCol` are used as array indices (`g.rows[g.cursorRow]`) and as pixel coordinates without verifying they fall within the grid's actual dimensions. An out-of-bounds cursor position from a corrupted snapshot would cause `cursorRow` to index into undefined array entries. The optional chaining on line 100 (`cursorRow?.[g.cursorCol]`) partially mitigates this, but the cursor rectangle would still be drawn off-canvas at `g.cursorCol * cellW`. **Recommendation**: Clamp `cursorRow` and `cursorCol` to `[0, g.rows.length - 1]` and `[0, g.cols - 1]` respectively before use.

- **`src/lib/key-to-ansi.ts:60-62` -- Alt+key passes arbitrary single characters to terminal without filtering**: The `Alt+key` branch sends `\x1b` + `e.key` for any single-character key. While this is correct ANSI behavior, if `onWrite` is ever exposed beyond `writePty` (e.g., to a plugin system or WebSocket), this becomes a vector for injecting arbitrary escape sequences into terminal sessions. **Recommendation**: This is acceptable for the current architecture where `onWrite` maps to Tauri's `write_pty` IPC command, but document the assumption that `onWrite` must only target the local PTY. If the output destination ever changes, add an escape-sequence allowlist.

- **`src/components/CanvasTerminal.tsx:121-123` -- Cell color values applied without validation**: `cell.bg` and `cell.fg` from the `GridSnapshot` are assigned directly to `ctx.fillStyle` and `ctx.fillStyle`/`ctx.strokeStyle`. Canvas2D is tolerant of invalid color strings (it silently ignores them), so this is not exploitable, but a malformed color value could cause rendering to silently break. **Recommendation**: Consider adding lightweight color validation (e.g., starts with `#` and is 4 or 7 characters) on the Rust side before emitting snapshots, as defense-in-depth.

- **`src/components/CanvasTerminal.tsx:203-210` -- Blink timer not gated on component visibility**: The `setInterval` for cursor blinking runs continuously at 530ms regardless of whether the terminal is visible (e.g., when the tab is backgrounded or the pane is not active). This is a minor performance/battery concern, not a security issue, but it does add unnecessary CPU wake-ups. **Recommendation**: Consider using `requestAnimationFrame` or pausing the timer when the component is not visible (via `document.visibilityState` or an IntersectionObserver).

## Nitpicks

- **`src/components/CanvasTerminal.tsx:233` -- `tabIndex={0}` without `role` attribute**: The container div is focusable but has no ARIA role. Screen readers and accessibility tools may not understand it is a terminal. This is not a security issue but could affect accessibility compliance. Consider adding `role="textbox"` or `role="application"` with an appropriate `aria-label`.

- **`src/lib/key-to-ansi.ts:4` -- `SPECIAL_KEYS` map uses `Record<string, string>` instead of a const object**: The type `Record<string, string>` allows arbitrary key access without type narrowing. Using `as const satisfies Record<string, string>` would provide stricter typing and prevent accidental mutation, consistent with the pattern used in `types.ts` for `EFFECT_MULTIPLIERS`.

- **`src/components/CanvasTerminal.tsx:109` -- `draw` callback missing `lineHeight` dependency**: The `draw` function's `useCallback` dependency array is `[background, cursorColor, fontSize, fontFamily]` but the function also uses `fontSize * lineHeight * dpr` indirectly through `cellMetrics.current`. While `lineHeight` affects `cellMetrics` via `measureCell`, a change to `lineHeight` without a resize event could cause stale cell metrics to be used during draw. Not a security issue but could cause subtle rendering bugs.
