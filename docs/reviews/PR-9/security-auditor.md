# Security Audit: PR #9 â€” IPC Adapter Layer

## Summary

This PR introduces a typed IPC adapter layer (`tauri-api.ts`) bridging the Tauri backend and React frontend, adds shared type definitions, wires `window.api` as a global, and adds the `tauri-plugin-shell` dependency scoped to `shell:allow-open`. The capability scope is appropriately narrow and the CSP is already configured. The main security concerns are: (1) the `openExternal` function passes URLs to the OS shell opener without any frontend validation or allowlist, creating a potential scheme-injection vector; (2) the `createPty` command forwards a `cwd` path and `startupCommand` to the backend with no frontend-side validation; and (3) several event listener patterns have a race condition where events can fire before the unlisten handle is stored, preventing cleanup.

## Must Fix

- **`src/lib/tauri-api.ts:16` -- `openExternal` passes arbitrary URLs to `shell:open` with no validation.** The `open(url)` call from `@tauri-apps/plugin-shell` hands any string directly to the OS default handler. A malicious or compromised frontend component could pass `file:///etc/passwd`, `ssh://`, `tel:`, or other dangerous URI schemes. There is no protocol allowlist enforced at either the frontend or capability layer. The Tauri `shell:allow-open` permission does not restrict which URL schemes are accepted. Add a frontend validation function that rejects anything outside `https://` (and optionally `http://`) before calling `open()`, or configure a Tauri shell scope with an explicit URL regex in `default.json` to restrict allowed schemes at the capability level.

- **`src/lib/tauri-api.ts:18` -- Silent error swallowing on `logScrollDebug`.** The `.catch(() => {})` silently discards all errors, including potential IPC failures that could indicate a compromised or malfunctioning backend channel. While this is a debug/telemetry call, silently swallowing errors in IPC communication makes it impossible to detect if the IPC channel is degraded. At minimum, log to `console.warn` in development mode so failures are observable.

- **`src/main.tsx:7` -- `window.api` global exposes full IPC surface to any script context.** Assigning the entire API object (including `writePty`, `createPty`, `killPty`, `openExternal`) to `window.api` means any script that runs in the webview (including anything injected via a future XSS) has unrestricted access to the PTY and shell APIs. While the CSP currently mitigates script injection, this is a defense-in-depth concern. If the CSP is ever relaxed (e.g., for a plugin system), the entire backend is exposed. Consider using module imports directly rather than a global, or at minimum freeze the object with `Object.freeze(api)` to prevent tampering.

## Suggestions

- **`src/lib/tauri-api.ts:31` -- `createPty` accepts unchecked `cwd` and `startupCommand`.** The `cwd` parameter is a file path and `startupCommand` is a shell command string, both passed directly to `invoke("create_pty", ...)`. While the Rust backend is the appropriate place for validation (and the backend commands don't exist yet in this PR), the frontend should still apply basic sanity checks: validate that `cwd` is a non-empty string that looks like an absolute path, and that `startupCommand` does not contain null bytes or other control characters. This provides defense-in-depth against logic bugs in callers.

- **`src/lib/tauri-api.ts:38-48,52-62,64-74,76-86,88-98` -- Event listener race condition allows missed unsubscribe.** All five event listener wrappers (`onTerminalRender`, `onPtyExit`, `onPtyCwdChanged`, `onPtyBranchChanged`, `onPtyPrChanged`) share the same pattern where `listen()` returns a `Promise<UnlistenFn>` but the returned cleanup function checks `unlisten` synchronously. If the caller invokes the cleanup function before the `listen()` promise resolves, the unlisten handle is lost and the listener leaks permanently. This is a resource leak that could also be a security concern if stale callbacks continue processing events after a component unmounts (e.g., rendering attacker-controlled terminal data into a detached DOM). Fix by storing the promise and awaiting it in the cleanup function, or by using an `AbortController` pattern.

- **`src/lib/tauri-api.ts:32` -- `writePty` sends raw string data to the PTY with no length limit.** There is no frontend-side size limit on the `data` parameter. A bug or malicious caller could send an extremely large string, potentially causing memory pressure in the IPC serialization layer or the Rust PTY handler. Consider adding a maximum payload size check (e.g., 1MB) before invoking.

- **`src-tauri/capabilities/default.json:6` -- Consider tightening shell scope with URL regex.** The `shell:allow-open` permission grants the ability to open any URL/path via the OS default handler. Tauri v2 supports scoped shell permissions with URL pattern matching. Adding a scope like `{ "open": { "url": true, "validators": [{ "url": "https?://.*" }] } }` would restrict `open()` to HTTP/HTTPS URLs at the capability layer, providing backend-enforced protection independent of frontend validation.

- **`src/shared/types.ts:193` -- `ScrollDebugEvent.details` accepts `Record<string, unknown>`.** This open-ended type is serialized and sent over IPC via `logScrollDebug`. If this data is ever logged to a file or displayed in a UI, the `unknown` values could contain deeply nested objects or circular references, causing serialization issues or log injection. Consider constraining the value type to primitives (string, number, boolean, null).

## Nitpicks

- **`src/shared/types.ts:94-95` -- `gradient` field accepts arbitrary CSS string.** The `gradient` field in `PaneTheme` is typed as `string` and documented as a "CSS gradient overlay." If this value ever originates from user input or a theme file, it could be used to inject CSS (e.g., via `url()` references). This is low risk since it appears to be internally generated, but adding a comment noting it should never be set from untrusted input would be helpful.

- **`src/vite-env.d.ts:5-8` -- Global augmentation is clean but lacks a `readonly` modifier.** The `Window.api` declaration allows reassignment (`window.api = maliciousApi`). While `Object.freeze` on the runtime value is the real protection, marking the type as `readonly api: KnkodeApi` signals intent and causes TypeScript to flag accidental reassignment.

- **`src-tauri/Cargo.toml:15` -- `tauri-plugin-shell` version is unpinned (`"2"`).** The dependency uses a major-version-only specifier. While Cargo's semver resolution is generally safe, pinning to a more specific range (e.g., `"2.3"`) reduces the risk of pulling in a new minor version with unexpected behavior changes. The frontend package.json already pins to `^2.3.5` which is more precise.

- **`src/lib/tauri-api.ts:15` -- `getHomeDir` exposes the user's home directory path.** This is a low-severity information disclosure concern. The home directory path is sent from the Rust backend to the frontend. If the frontend ever logs or transmits this value, it reveals the local username and OS. This is expected for a terminal application but worth noting for awareness.
