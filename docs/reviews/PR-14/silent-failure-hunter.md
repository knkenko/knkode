# PR #14 Silent Failure Audit: Config Store Hardening

## Summary

The PR introduces meaningful hardening to config file I/O (corrupt JSON recovery, theme migration, snippet validation, Unix file permissions), but several error paths silently swallow failures or silently alter user data without any indication to the caller or user. The corrupt-JSON recovery path is the most concerning: it converts parse errors into `Ok(None)`, which downstream callers interpret as "file does not exist," causing permanent data loss when the original file is then overwritten on the next save.

## Must Fix

- **`src-tauri/src/config.rs:424-438` -- Corrupt JSON silently treated as "file not found"**: When `serde_json::from_str` fails, the function logs to stderr and returns `Ok(None)`. Callers (`read_json_array` at line 443, `read_json_object` at line 452) treat `None` identically to "file does not exist" and return an empty array/object. The very next `save_workspace` or `save_snippets` call will then overwrite the corrupt file with an empty or partial dataset. The `.corrupt` backup exists, but the user has no idea their data was just discarded. The `Result` return type is being misused here -- a parse failure is being reported as success. **This should return an `Err` that propagates to the frontend so the user can be told "Your config file was corrupt; a backup was saved to X. The app is starting with defaults." At minimum, the error must be distinguishable from "file not found" by callers.**

- **`src-tauri/src/config.rs:433-435` -- Backup failure silently ignored**: If `fs::copy` fails when creating the `.corrupt` backup, the error is printed to stderr and execution continues, returning `Ok(None)`. This means: (1) the corrupt file was not backed up, (2) the caller gets empty defaults, (3) the next save overwrites the corrupt file, destroying the only copy of the user's data forever. If the backup fails, this must return `Err` so the data destruction chain is halted.

- **`src-tauri/src/config.rs:386-389` -- Snippet filtering silently drops entries with no feedback**: `get_snippets` silently filters out any snippet that fails `is_valid_snippet`. If a user has 20 snippets and one has a missing `command` field (perhaps due to a frontend bug or hand-edit), they will see 19 snippets with no indication that one was dropped. Since snippets may contain valuable user-authored data, silently dropping them is data loss. At a minimum, log which snippets were dropped (by id if available) and how many. Ideally, return the count of dropped snippets alongside the valid ones so the frontend can show a warning.

- **`src-tauri/src/config.rs:328-331` -- Migration runs on every read with no persistence or notification**: `get_workspaces` applies `migrate_workspace` to every workspace on every read, but never writes the migrated data back. This means: (1) migration runs on every single call, which is wasteful, (2) if migration has a subtle bug, it silently corrupts workspace data on every read without any log entry, (3) the user has no idea their theme data was transformed. Migrations should log when they actually transform data (at least once), and ideally the migrated result should be persisted so migration is a one-time operation.

## Suggestions

- **`src-tauri/src/config.rs:481-486` -- File permission failure logged to stderr only**: After a successful atomic write, if `set_permissions` fails, the error is printed to stderr and `Ok(())` is returned. The caller (and user) believes the write succeeded with proper permissions. On a shared system or when config contains sensitive data, a permission failure may be security-relevant. Consider returning `Err` here, or at least documenting explicitly why this is acceptable to ignore. The directory permission set at line 317-318 correctly uses `map_err` and propagates the error -- the file permission handling should follow the same pattern for consistency.

- **`src-tauri/src/config.rs:144-261` -- `sanitize_theme` silently replaces invalid values with defaults, no log**: When a user's hand-edited theme has an invalid hex color, bad font size, or unrecognized cursor style, the value is silently replaced with a default or silently dropped. There is no log entry and no way for the user to know their customization was discarded. For example, if a user sets `"background": "rgb(26,26,46)"` (valid CSS but not hex), it will be silently replaced with `#1a1a2e`. A debug-level log for each replaced field would make this diagnosable.

- **`src-tauri/src/config.rs:62-78` -- `migrate_theme` replaces entire theme block silently when theme is non-object**: If `ws["theme"]` exists but is not an object (e.g., `"theme": "dark"` from a hypothetical future schema), the entire value is silently replaced with a default theme object. No log, no indication to the user. This should at minimum log when it replaces an existing non-null theme value.

- **`src-tauri/src/config.rs:104-107` -- `migrate_effect_levels` silently returns unmodified workspace when theme is missing**: If a workspace has no `theme` key at all, the function returns early with no migration and no log. Combined with the fact that `migrate_theme` would have already injected a default theme, this is likely fine in practice, but the silent early return makes the migration pipeline fragile -- if the function ordering in `migrate_workspace` changes, this could silently skip migration.

## Nitpicks

- **`src-tauri/src/config.rs:427-430` -- `eprintln!` is not a structured logging mechanism**: All error logging in this file uses `eprintln!`. This goes to stderr which is typically not captured in production Tauri apps (stderr is not visible to end users or sent to any error tracking). Consider using the `log` or `tracing` crate so these messages are captured by Tauri's logging infrastructure and potentially surfaced to crash reporting. The existing codebase uses `eprintln!` elsewhere too (e.g., `pty.rs`, `terminal.rs`), so this is a project-wide concern, but the config store errors are more consequential because they involve data loss.

- **`src-tauri/src/config.rs:249-258` -- AnsiColors validation is all-or-nothing**: If 15 of 16 ANSI color keys are valid hex and one is corrupt, the entire `ansiColors` block is silently dropped. A partial validation that preserves valid entries and replaces invalid ones with defaults would be more user-friendly.

- **`src-tauri/src/config.rs:392-395` -- `save_snippets` does not validate before writing**: While `get_snippets` validates on read, `save_snippets` accepts any `Vec<Value>` and writes it directly. A subsequent `get_snippets` call would then silently filter out anything invalid that was just saved. Validating on write (or at least logging) would prevent the confusing "I saved it but it disappeared" experience.
